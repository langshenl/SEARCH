# Browser Click Search

## Overview

把“搜索”从轻量 HTTP 抓取切换到更接近真人的浏览器交互模式。

核心用途：
- 搜索引擎返回安全验证、challenge、anomaly 页面时兜底
- 搜索结果页 HTML 结构变化，导致正则解析失效时兜底
- 需要通过真实页面渲染、点击、跳转来拿到搜索结果或真实落地链接时使用

这个技能是**兜底搜索技能**，不是默认第一选择。默认仍应优先尝试轻量搜索；当轻量搜索失效，再切到本技能。

## Trigger Conditions

在以下情况优先考虑本技能：

1. 轻量搜索返回 0 条，但直觉上不应该是 0 条
2. 搜索引擎页面返回安全验证、验证码、challenge、anomaly
3. 搜索页 HTML 能打开，但脚本解析规则明显失效
4. 用户明确要求“模拟点击”“像真人一样打开百度搜一下”“浏览器方式搜”
5. 需要打开结果页并拿真实落地链接，而不是只拿搜索页跳转链接

## Workflow

### 1. 明确搜索词

先确定本次要搜的原始搜索词。

优先策略：
- 先用自然搜索词
- 默认不强制加 `site:`
- 如果用户明确要求限定官网域名，再追加 `site:` 版本

示例：
- `湖北省政策`
- `河北省博物馆政策`
- `全国博物馆实施意见`

### 2. 打开搜索页

优先使用真实浏览器环境打开搜索引擎页面，而不是仅用 `urllib` / `requests` 拉 HTML。

推荐顺序：
1. 百度
2. Bing
3. 其他可用搜索入口

要求：
- 等待页面完成初步渲染
- 识别是否出现验证码、安全验证、异常流量页
- 如果出现验证页，明确记录“被风控”，不要误报成“无结果”

### 3. 提取可见结果

从浏览器当前可见结果中提取：
- 标题
- 搜索结果链接
- 摘要
- 搜索引擎来源

如果能直接拿到真实落地链接，优先保存真实落地链接。

如果页面中的链接是搜索引擎跳转链接：
- 可先保留跳转链接
- 如有必要，再点击结果并记录最终落地链接

### 4. 模拟点击（按需）

只有在以下情况才执行点击：
- 需要确认真实落地页 URL
- 搜索页不直接暴露真实链接
- 需要验证目标页是否真的是政策页 / 官方页 / 文件页

点击策略：
- 不要无节制乱点
- 每轮优先点前几条最像官方结果的链接
- 点击后记录最终跳转地址，再回退继续采集

### 5. 输出结构

最终输出仍应统一成轻量结构，便于后续清洗：
- title
- url
- snippet
- engine

如果拿到了额外字段，也可以附加：
- final_url
- visible_rank
- source_page_url
- blocked_reason

## Operational Rules

- 本技能是搜索执行层兜底，不负责政策扩词和最终去重清洗。
- 如果用户只要求“模拟点击搜一下”，可以直接用原始搜索词执行，不必先扩词。
- 如果后续要并入政策搜索主流程，应作为“轻量搜索失败后的二级方案”，而不是替代全部轻量搜索。
- 遇到风控页、challenge 页时，要明确标记为“blocked / challenged”，不要假装是正常 0 结果。
- 若要用于政策/官方文件搜索，后续仍建议把结果送入 `policy-dedup-curator` 做清洗。

## Relationship To Other Skills

- 与 `policy-search-fixed-workflow` 的关系：
  - `policy-search-fixed-workflow` 负责主流程编排
  - 本技能负责搜索执行层的浏览器兜底方案

- 与 `policy-dedup-curator` 的关系：
  - 本技能负责拿搜索结果
  - `policy-dedup-curator` 负责结果清洗、去重、官方优先整理

## Next Implementation Direction

当前先把规则固化为技能，后续真正落地浏览器执行时，优先做这几件事：

1. 选择浏览器自动化方案
2. 先实现“打开搜索页 + 判断是否被风控”
3. 再实现“提取可见搜索结果”
4. 最后再加“按需点击并回填真实落地链接”

不要一开始就把点击、翻页、深度抓取全部堆上去；先做最小可用版本。
