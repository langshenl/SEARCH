# Policy Search Workflow Memory

按这套固定流程理解和执行政策搜索请求，除非用户明确要求改流程。

## 1. 当前系统结构（最新）

当前主流程已经切换为**轻量三段式**，不要再默认走旧的 policy_harvest_system / search_policy_fast 双层脚本链路。

当前实际主链路：
- 扩词技能：
  - `~/.openclaw/workspace-search/skills/general-policy-expand/`
- 搜索执行脚本：
  - `~/Desktop/通用搜索脚本.py`
- 去重清洗技能：
  - `~/.openclaw/workspace-search/skills/policy-dedup-curator/`

也就是说：
- 当前日常政策搜索，优先走 `general-policy-expand -> 通用搜索脚本 -> policy-dedup-curator`
- 不要再默认使用 `~/Desktop/政策采集系统/policy_harvest_system.py`
- 也不要再默认依赖已移除的 `policy-search/search_policy_fast.py`

## 2. 核心原则

- 用户只提供自然语言需求，不应被要求手动扩词、手动逐条搜索、手动去重。
- 默认由当前对话内代理直接执行，不创建子 agent。
- 固定流程是：
  1. 解析自然语言
  2. 先把扩词内容返回给用户
  3. 再逐条调用桌面通用搜索脚本
  4. 原始结果统一保存到桌面搜索数据文件夹
  5. 搜索完成后再做去重清洗
  6. 默认输出混合版和官方优先版 Excel 到桌面政策库文件夹

## 3. 解析规则

- 没写年份 → 默认当前年份
- 没写地区 → 默认湖北省
- 写了“全国” → 全国模式
- 写了具体省份/自治区/直辖市 → 单省模式

## 4. 扩词与执行格式（硬规则）

当前主流程不再强依赖 `current-expansion-plan.csv`。

日常执行时，必须先在对话中明确返回本次扩词结果，再逐条执行搜索脚本。

单次政策搜索默认采用：
- **保留原始输入词 1 条**
- **再补 4 条更稳的政策标题型扩词**

也就是通常总共 5 条，例如：
1. 湖北省 政策 site:hubei.gov.cn
2. 湖北省 发展规划 site:hubei.gov.cn
3. 湖北省 实施意见 site:hubei.gov.cn
4. 湖北省 若干措施 site:hubei.gov.cn
5. 湖北省 管理办法 site:hubei.gov.cn

其中：
- 只要是**政策相关搜索**，扩词必须带对应域名锚点，即 `site:{host}`
- 普通搜索不强制带 `site:`
- 如果是明确主题政策，则把“政策 / 发展规划 / 实施意见 / 管理办法 / 若干措施”替换为主题相关的对应方向词

## 5. 范围规则

- 全国模式：按省份展开，**每省固定 3 行**
- 单省模式：通常固定 20 行
- 但如果用户没有提及具体政策主题，只是搜索“某省政策内容 / 某省相关政策内容”，则**不要硬凑 20 条**，直接走 8 大核心领域：
  1. 粮食安全政策
  2. 耕地保护政策
  3. 产业发展政策
  4. 绿色生态政策
  5. 科技装备政策
  6. 经营主体政策
  7. 金融保险政策
  8. 乡村建设政策

这条规则优先级高于“单省固定20行”。

## 6. 扩词质量规则

### 普通主题模式

例如：农业、博物馆、教育、新能源、道路。

- 命中模板 → 用模板扩展
- 未命中模板 → 用通用方向扩展
- 但普通主题扩词必须优先使用更像真实政府文件标题的词，例如：
  - 发展规划
  - 实施意见
  - 若干措施
  - 管理办法
  - 建设方案
  - 专项规划
  - 工作通知
  - 申报指南
- 泛方向词（如提升、保障、监管、创新）可以保留，但应后置，不要默认排在最前面。

### 全量政策模式

- 不能直接使用一级分类裸词作为搜索词，例如：经济政策、科技政策、工业政策、招商投资政策。
- 单省模式应直接生成可检索执行词，或在泛政策内容场景下直接使用 8 个核心方向。

## 7. 主题清洗规则

扩词脚本必须先做主题标准化，清理口语残片，例如：
- 帮我搜索一下 / 帮我搜一下 / 搜索一下 / 搜一下 / 搜索 / 搜
- 相关内容 / 政策相关内容 / 相关 / 内容
- 帮我 / 一下 / 的 / 方面 / 领域

例如：
- `搜索一下广东的博物馆相关内容` → 应标准化为 `博物馆政策`

不能再生成类似：
- `搜索一下博物馆内容政策`

## 8. 搜索执行规则（最新）

当前执行脚本：
- `~/Desktop/通用搜索脚本.py`

执行层必须：
- 严格按已经返回给用户的扩词逐条执行
- 同一次任务的所有原始结果，统一存到桌面同一个任务目录，不再创建 `01/02/03` 之类的编号子目录
- 推荐任务目录格式：
  - `~/Desktop/搜索数据文件夹/YYYYMMDD_任务名/`
- 当前搜索引擎结果不是硬域名过滤，`site:` 只是强约束搜索词；真正的官方优先整理放到清洗层完成

## 9. 单行执行性能规则

主脚本单行执行默认采用：
- **轻量 3 轮**
  1. 主站搜索
  2. 部门补搜
  3. 目录补搜

并启用早停：
- 当前行已经召回到结果 → 提前结束本行
- 连续 3 轮都是 0 结果 → 提前结束本行

不要把地市补搜 / 专题补搜当作每行默认必跑项。

## 10. 召回数量规则（最新）

- **不要再把单轮搜索结果数人为卡成很小固定值（例如 4 条、10 条）**。
- 搜索执行层应遵循：**能拿到多少数据就尽量拿多少**。
- 需要特别检查底层搜索函数（如 `ddg_html_search` / `bing_search`）的默认 `max_results`，避免上层放开后下层仍然偷偷卡成 10。
- 性能优化应优先通过“轻量轮次 + 早停”解决，而不是通过把单轮召回条数压得过低来解决。

## 11. 清洗输出规则（最新）

搜索完成后，必须再走 `policy-dedup-curator` 清洗层。

硬规则：
- 必须同时具备：**标题、摘要、原文链接**
- 缺任一项直接丢弃
- 以原文链接做 URL 去重

默认输出两版：
- **混合版**：去重后的完整结果
- **官方优先版**：优先保留政府官网、委办局官网、官方解读/政策页结果

输出目录固定为：
- `~/Desktop/政策库文件夹/`

## 12. 目录规则（最新）

### 最外层运行目录

原始数据和最终数据的最外层目录，不再只用年份。
必须使用：
- `年份-时分秒`

例如：
- `2026-201530`

### 原始数据目录

- 单省模式：
  - `~/Desktop/原始数据/{year-hhmmss}/{省份}/`
- 全国模式：
  - `~/Desktop/原始数据/{year-hhmmss}/全国/{省份}/`

重点：
- **全国模式下必须先创建“全国”文件夹，再在里面按省份细分**
- 不允许把全国模式的各省结果直接平铺在最外层运行目录下面

### 最终数据目录

- 单省模式：
  - `~/Desktop/最终数据/{year-hhmmss}/{省份}/`
- 全国模式：
  - `~/Desktop/最终数据/{year-hhmmss}/全国/`

## 13. 路径记忆

- 扩词输出表（如脚本直接导出时）：`~/Desktop/搜索配置文件夹/general-policy-expand-output.csv`
- 原始搜索结果目录：`~/Desktop/搜索数据文件夹/`
- 去重清洗结果目录：`~/Desktop/政策库文件夹/`
- 搜索执行脚本：`~/Desktop/通用搜索脚本.py`

## 14. 一句话总流程

用户给自然语言需求 → `general-policy-expand` 先返回带域名锚点的扩词 → 按扩词逐条调用 `~/Desktop/通用搜索脚本.py` → 原始结果统一写入 `~/Desktop/搜索数据文件夹/任务目录/` → 再用 `policy-dedup-curator` 做去重清洗 → 默认输出混合版和官方优先版 Excel 到 `~/Desktop/政策库文件夹/`。
